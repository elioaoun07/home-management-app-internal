✅ 2026-07-16 — **DW-1: Flight recorder foundation.** Full-fidelity transcript capture + v2 usage/cost, layered onto the existing S1–S3 Delivery runner without changing any state-machine transition or gate semantics.

Side-fix: `emitEvent`'s O(n²) full-ndjson reparse on every append (an unrelated pre-existing inefficiency, `events.mjs`) replaced with an in-memory per-session seq cache — output format unchanged.
- `scripts/delivery/drivers/fake.mjs` (extended) — seam v2 surface for tests: pure-data `manifest()`, `onRaw` full-fidelity record feed, distinct `usageV2` passthrough (kept separate from the always-v1-shaped `usage` field, matching the real-driver contract), per-turn model/effort tracking onto the ref, default `turnMeta`.

✅ 2026-07-16 — **DW-8: Provider handoff (Claude ↔ Codex).** Switching providers mid-Delivery without silent context loss.

✅ 2026-07-16 — **DW-9: Fork + lineage.** Branch an independent session from any checkpoint.

✅ 2026-07-16 — **DW-10: Mid-turn abort (gated).** The final slice — explicit, owner-initiated, lost-work-acknowledged abort of a turn already in flight.

- A turn is a single `await` inside one `advanceSession` tick, and controls are only drained *between* ticks — so reaching a turn that's already running required a genuinely concurrent mechanism: `watchForAbort()` in `run-session.mjs` polls `controls/` (default every 300ms, configurable) for a **new** `pause{abortInFlight:true}` control while `driver.runTurn()` is in flight, and calls `AbortController.abort()` the moment one appears. The poller only detects and aborts; the control itself is still drained normally by `drainControls()` on the *next* tick (single-writer discipline preserved).
- `driver.mjs` gained `DriverAbortedError` (a distinct, non-retried outcome). `fake.mjs` implements a real abortable delayed turn (`delayMs` + signal race) for offline testing. `claude.mjs` bridges the seam's `signal` onto the SDK's `Options.abortController` (`bridgeAbortSignal()`); `codex.mjs` passes `signal` straight through to `TurnOptions` (`withAbortSignal()`) — both per the SDK types verified in the plan; live-SDK abort behavior itself remains unverified (same provider-turn approval gate as every other live-only check in this program).
- An aborted turn is sealed `result:"aborted"` in `transcript/turns.ndjson` (never retried, unlike a generic failure) and always runs the git-guard + a **workspace delta** (`workspaceDelta.changedPaths`) — the response is lost outright, but the workspace is never rolled back, so the owner needs to see exactly what an aborted BUILDING turn touched. `blockFrom()` renders this as an explicit "Turn aborted by owner. The workspace was not rolled back — N file(s) changed... review the diff (or run validation) before retrying" message, surfaced in both the BLOCKED gate panel and the Timeline's red card.
- `SessionDetail.jsx` gained an **Abort turn…** button (gated on the current provider's manifest `supportsAbort`, hidden while paused/terminal) opening an explicit confirm dialog with the full lost-turn/no-rollback copy before posting the abort control.
- **Tests:** `run-session-abort.test.ts` — genuinely races a real, artificially-delayed fake-driver turn against a control file written mid-flight (real timers), asserting the turn seals as `aborted`, the phase lands on BLOCKED with the right message, the abort is never retried, and the pending pause control drains correctly on the following tick. Plus abort-signal unit tests in `drivers.test.ts`/`drivers-claude.test.ts`/`drivers-codex.test.ts`. Live-verified: a parked-at-BUILDING fixture to check the button/dialog copy, and a pre-aborted fixture to check the BLOCKED/Timeline/Conversation rendering — both fixture sessions deleted after verification.

- **Per-phase driver mode (the actual killer):** `getHandle()` in `run-session.mjs` now passes the requested `mode` into `driver.resume(ref, {mode})` every turn instead of a frozen `ref.mode` established once in DISCOVERY — BUD-11's BUILDING turns ran read-only for its entire 2.5h lifetime (`driver.ref.mode:"readonly"`, never updated), so 10 "build steps" wrote prose into `build-log.md` and never touched the target test file. `resume()` in `claude.mjs`, `codex.mjs`, and `fake.mjs` all accept a per-call `overrides` param now; `ref` itself is never mutated.

- **Transcript completeness:** `onRaw` is now wired in `claude.mjs`'s and `codex.mjs`'s real `runTurn()` (only `fake.mjs` fed it before) — assistant text, reasoning, full tool inputs/outputs now persist per-turn in the DW-1 shard format, not just the 2000-char-truncated `events.ndjson` timeline.

✅ 2026-07-22 — **DW-14: Preflight timeout/abort (defensive — did NOT resolve the reported incident).** While triaging a "session stuck at `SELECTED` with no error" report I first hypothesized the hang was inside `driver.startSession()` → `preflight()` (the one call before any turn, before `runGuardedTurn`'s abort machinery), because the session sat at `SELECTED`/`decisionsProcessed:0` with a stale heartbeat. That hypothesis was **wrong** — `preflight()` only runs in DISCOVERY, and these sessions never left SELECTED (DW-15 has the true cause). The preflight timeout is still a legitimate hardening (an unbounded `preflight()` genuinely *could* hang with no recovery), so it ships — but it is not what the owner was hitting. Honest note kept here so the next reader doesn't mistake it for the fix.

- **`claude.mjs`:** `preflight()`'s `for await` over `query()` now races a 30s timeout (`PREFLIGHT_TIMEOUT_MS`, exported) that aborts a purpose-built local `AbortController` (wired into `Options.abortController`) and rejects into the existing `authentication preflight failed` error path — unchanged for every already-passing case, new only for the hang case.
- **`codex.mjs`:** the analogous `thread.run("Reply with exactly: OK")` in `startSession()` had the identical unbounded-hang shape (no timeout, no signal) — fixed the same way, passing `{signal}` (the SDK's real `TurnOptions.signal`, verified against the installed `@openai/codex-sdk` `.d.ts`) and racing the same 30s timeout.
- **`driver.mjs`:** new shared `withTimeout(promise, ms, onTimeout)` helper (both drivers import it) — races a promise against `ms`, calls `onTimeout()` to abort the in-flight call, then rejects with a `DriverError` instead of hanging.
- A preflight timeout surfaces like any other preflight failure: `getGuardedHandle()` catches it and `blockFrom()` durably transitions to BLOCKED — recoverable from the dashboard instead of a manual process kill.
- **Tests:** 2 regression cases (`drivers-claude.test.ts`, `drivers-codex.test.ts`) using `vi.useFakeTimers()` against a never-resolving driver call, asserting the timeout fires and the SDK abort signal is set.

✅ 2026-07-22 — **DW-15: The actual "silently stuck at SELECTED" root cause — event-loop-blocking validation baseline.** After DW-14 shipped, the owner reported the session *still* hung with no error. Deeper forensics (Windows `Get-CimInstance` process tree, not just PID liveness) showed the runner processes each had a live child: `cmd.exe /d /s /c "pnpm lint"`. The DW-12 validation baseline (`handleSelected`, gated on `dirtyAtStart`) ran `pnpm typecheck`+`lint`+`test` **synchronously via `spawnSync`**, which blocks the entire Node event loop. Recorded baseline from `s-20260722-203135-cv12`: `typecheck` 13.7 s (ok), then `lint` **646.8 s (~11 min)** before it was killed. The full failure chain:

  1. Dirty-tree session → `handleSelected` runs the baseline **synchronously** → event loop frozen for the whole ~11 min.
  2. Frozen loop → the heartbeat timer can't fire → `runner.json` goes stale within 15 s → dashboard reports the runner dead (`isRunnerAlive` hits the `age > staleMs` branch *before* the pid probe, so a blocked-but-alive process reads as dead).
  3. Owner clicks **Resume** → the stale-heartbeat guard is fooled → a **second** runner spawns → runs the same 11-min suite **concurrently**. Each Resume made it worse. No error ever surfaced because nothing crashed — it was silently grinding a doubled 11-min lint.
  - This is also why DW-14's `setTimeout`-based preflight timeout could never have helped even if preflight *were* the culprit: a synchronously-blocked event loop cannot fire timers.

- **Async, tree-killing, timeout-bounded validation (`run-session.mjs`):** `runValidationCommands` is now `async`, running each command as a real `spawn`ed child (event loop stays free) with a per-command timeout (`config.budgets.validationTimeoutMs`, default 240 s). On timeout the **whole process tree** is killed — `killProcessTree()` uses `taskkill /pid /T /F` on Windows / a process-group `SIGKILL` on POSIX (`detached` there), because a plain `child.kill()` only signals the `cmd.exe`/`pnpm` wrapper while the real `tsc`/`eslint`/`vitest` grandchild keeps running and holds the stdio pipes open (measured: a 3 s timeout still took 11.7 s to return before the tree-kill; 3.3 s after). Covers both the baseline (`handleSelected`) and post-build (`handleValidating`, now `async` + awaited) paths in one change. A timed-out command is recorded `timedOut:true` with an annotated excerpt.
- **Interval heartbeat (`runLoop`):** the heartbeat was only written *between* ticks, so any single long tick (the baseline) went stale. `runLoop` now drives a `setInterval` heartbeat that beats every `heartbeatIntervalMs` regardless of tick progress — with async validation the loop is free so it fires, keeping `isRunnerAlive` fresh and making the resume guard correctly refuse a duplicate runner. This is the fix that breaks the double-runner cascade at its source.
- **Config:** `budgets.validationTimeoutMs` (default 240 s) added to `config.mjs`.
- **Transparency:** a `validation.baseline.started` event is emitted before the (possibly multi-minute) baseline so the timeline explains the wait instead of looking dead.
- **Tests:** `run-session.test.ts` — async-seam rewrite of the 3 `runValidationCommands` unit tests + a new timed-out-command case; a new `runLoop` regression asserting the heartbeat beats on its interval independent of per-tick work. Full `tests/delivery` green (883 passing). Real-environment verified on this Windows host: `runValidationCommands` with a 3 s timeout against the live repo returned in 3.3 s with `timedOut:true` and **no orphaned `tsc` process** (tree-kill confirmed).
- **Separate finding (NOT a delivery-feature bug, flagged for the owner):** `pnpm lint` itself takes ~11 min because the flat `eslint.config.mjs` has no `ignores`/`files` scoping, so bare `eslint` type-checks the whole repo (`scripts/pm/` — a separate app, `tests/`, …). `eslint src` alone is 13 s. ✅ 2026-07-22 — **DW-17: Runner crash containment + dead-runner observability — the "stuck session shows me nothing" fix.** Reported live against `s-20260722-221533-wous` ("sessions seem stuck, nothing works"). Forensics found **two stacked problems**: (1) the 4-minute dirty-tree validation baseline (typecheck 15.5 s ok, lint killed at the 240 s bound) ran with zero visible progress, so the session *read* as stuck while healthy; then (2) the moment it transitioned to DISCOVERY, `getGuardedHandle`'s pre-turn `takeSnapshot(repoRoot)` — the one call sitting *outside* its try/catch — threw (`git status --porcelain --untracked-files=all` exited non-zero, transient: it doesn't reproduce; timing points at the lint-timeout `taskkill /T /F` and/or a concurrent git process racing the freshly spawned `git.exe`), the exception propagated through `handleDiscovery` → `advanceSession` → `runLoop` → `main().catch` and **killed the runner process**. - **Git errors now say why (`gitread.mjs`):** `runGitRead` catches `execFileSync` failures and re-throws a `GitReadError` carrying the exit code / killing signal **and git's stderr** (`git status … failed (exit 128): fatal: …`), with the original error as `cause`. The crash that started this incident logged a message that named nothing; the next one names the actual git failure.
- **Pre-turn snapshots can't kill the runner anymore (`run-session.mjs`):** both unprotected `takeSnapshot` call sites (`getGuardedHandle`, `runGuardedTurn`) now retry once after 1 s (transient-git-race tolerance), then fail like any other setup error — the phase handler routes it through `blockFrom()` → BLOCKED with the git error visible at the gate.
- **Last-resort containment (`runLoop`):** any unexpected throw out of `advanceSession` is caught by new `parkSessionOnCrash()` — re-reads state from disk (the handler may have persisted newer state before throwing), appends the stack to `runner.log`, emits a `runner.error` event, and parks the session `BLOCKED` via the universal `error.fatal` transition with `lastError.errorKind:"runner-crash"` — owner sees the cause and a Retry gate; the process only dies (rethrow) when the state machine has no legal block from the current state, and even then the error is on the event record first.
