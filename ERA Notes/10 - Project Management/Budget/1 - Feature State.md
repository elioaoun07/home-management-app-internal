# Budget · 1 — Feature State

> **Command Center:** [\_index](_index.md) · [1 · Feature State](<1 - Feature State.md>) · [2 · Vision & Roadmap](<2 - Vision & Roadmap.md>) · [3 · Action Plan](<3 - Action Plan.md>) · [4 · Checklist](<4 - Checklist.md>)
>
> **What this file is:** the _honest, no-hype_ state of every Budget sub-feature — what exists, how mature it is, and the single most useful next step. **No imagination here** (that's file 2).
>
> **Method & confidence:** a **structural** assessment derived from the cluster's vault docs (`02 - Standalone Modules/`), live route/API surface, and `src/features/`. It is **not** a line-by-line correctness audit. Treat tiers as "how battle-tested," not "bug-free."
>
> **Module identity:** "Budget" is the user-facing name for the **finance cluster** — several architecturally independent standalone modules (accounts, transactions, categories, recurring, transfers, allocation, statement import, analytics, debts, future purchases, drafts) that together form the money side of the app. At the app level (global [2 · Feature State](<../2 - Feature State — Current Reality.md>)) the core of it is **🟢 Core, stable** but **untested** — this is the financial heart and carries the highest regression cost.

## Implemented Notes

- [x] 2026-06-29 - **Transfer undo/edit balance-history audit no longer violates the DB constraint.** `account_balance_history.change_type` now accepts `transfer_deleted` and `transfer_updated`, matching the existing transfer delete/update API emitters, and transfer balance-history rows now carry `transfer_id` so the history drawer can resolve the related transfer. DELETE/PATCH now ignore already soft-deleted transfers before balance reversal so retries cannot reverse the same transfer twice. The same constraint migration also accepts the existing `statement_import` emitter. Migration: `migrations/2026-06-29_account-balance-history-change-types.sql`. Typecheck green.
- [x] 2026-06-27 - **NFC transfer modal supports multi-transfer sessions.** Modal no longer closes on confirmation — each successful transfer logs a green row (amount · from → to) and resets the amount field so the next transfer can be started immediately. Auto-advances to the logical next template after completion (salary-deposit → refill-wallet). Close button relabels to "Done" after the first transfer. Submit button renamed "Transfer". Dialog is wider (`sm:max-w-md`) with larger touch targets (chips `min-h-[2.5rem]`, selects `h-12`, amount `h-14`, buttons `h-12`) for mobile thumb-zone ergonomics.
- [x] 2026-06-27 - **NFC transfer modal now shows partner's public accounts.** When the partner taps the NFC tag, `NfcWalletTransferPrompt` was using `useMyAccounts()` (own-only) so she could only see her own accounts and couldn't select the owner's public accounts (e.g. shared Wallet). Fixed by switching to `useAccounts()`, which returns own accounts + household partner's `is_public` accounts — the same contract already used by the expense form's account picker. Template pre-fill still resolves correctly because own accounts sort first.
- [x] 2026-06-27 - **NFC Salary -> Wallet refill now resolves only the logged-in user's own accounts.** The shortcut prompt had drifted to `useAccounts()`, which includes partner-public shared accounts and could preselect/display the partner's Salary or Wallet on a physical NFC tap. `NfcWalletTransferPrompt` now uses `useMyAccounts()`, resolves optional `from`/`to` references by own account ID first and then own account name, and derives the select values directly from those validated matches so UUID URLs visibly preselect the correct rows. The general `TransferDialog` still supports shared accounts by design.
- [x] 2026-06-27 - **Budget AI dashboards now survive chat history reloads without another AI call.** The structured `AnalysisReport` was returned to the current client session but dropped when the assistant response was persisted to `ai_messages`, so reopening a previous conversation lost the **View as Dashboard** button and forced a full regeneration. Added `ai_messages.analysis_report` (`migrations/2026-06-27_ai-message-analysis-report.sql`, reflected in `schema.sql`), store the report JSON with the assistant row, select it in `GET /api/ai-chat`, and hydrate it back into `AIChatAssistant.loadConversation()`. Dashboard re-open now reuses saved JSON as a pure visual translation; only pre-migration answers lack stored reports. Typecheck green.
- [x] 2026-06-27 - **Budget AI dashboard now tolerates duplicate category labels.** React warned when a structured analysis returned repeated category names such as `Food`, because `AnalysisDashboard` keyed both the donut cells and ranked legend rows by `s.name`. The category widget now merges duplicate names case-insensitively before slicing the top categories, recomputes each displayed share from the merged total, preserves the first useful color/comment, blends MoM deltas by amount, and uses generated slice IDs for React/Recharts keys. This keeps the ephemeral dashboard stable even when Gemini repeats a human label. Typecheck green.
- [x] 2026-06-27 - **Budget AI upgraded from a one-line greeting into a deep spending-analysis engine with a one-tap dashboard view.** The floating Budget AI assistant (`AIChatAssistant`) returned short plain-text answers and rendered them with `whitespace-pre-wrap`, so `**bold**`/`##` showed as literal characters; output was capped at `maxOutputTokens: 1024` with no enforced structure (it could stop after a ~45-token greeting). Introduced a single strict `AnalysisReport` contract (`src/lib/ai/analysisReport.ts`): analysis-intent messages (or `mode:"analysis"`) now call Gemini with `responseMimeType:"application/json"` + `responseSchema` (temp 0.3, 8192 tokens) over **precomputed** metrics (category share, month-over-month delta, savings rate, a 6-month income/expense trend added to `fetchBudgetContext`), validated by a **tolerant Zod** schema with a **deterministic fallback** (`buildFallbackReport`) so a real, data-backed report always exists even if the model is rate-limited or returns junk. - [x] 2026-06-27 - **Budget "Spent" now uses the same account scope as the dashboard (hidden accounts excluded).** After drafts were excluded, the Budget card read `$1,871` vs the dashboard's correct `$1,856` (a $15 gap). - [x] 2026-06-27 - **Spending totals reconciled to the penny across Budget + Review V3 (canonical `sumSpending`).** Four "spent this month" figures disagreed — Budget Spent card $2,132 · Insight $1,856 · Monthly $2.1k · Categories $1.9k — because three data sources (budget API, analytics API, client transactions) each applied different rules. The Insight pie and the Monthly bars now both derive income/expense from the **same client transactions** via that rule; Categories already matched (the "$1.9k" was just `fmtDollar` rounding of $1,856). `GET /api/budget-allocations` now accepts a Zod-validated explicit `start`/`end` window, excludes debt-returns + partner-private, uses absolute amounts, and computes `total_spent`/`user_spent`/`partner_spent` **window-wide independent of category mapping** so uncategorized + hidden spend is included. `WebBudget` passes the custom-month window (from `useDatePreferences`) so the Budget page and the dashboard share one billing window; `useBudgetAllocations(month, accountId, range)` extended.
