## The decision: two-stage Capacitor on the existing codebase

### Stage 1 — remote shell (ship it)

Native Android + iOS projects generated by **Capacitor 7** whose WebView loads a **dedicated mobile production deployment** (`server.url` in `capacitor.config.ts` → the `era-mobile` Vercel project's domain), with the Capacitor native bridge injected into the remote page. Native capabilities come from plugins accessed through a new `src/lib/native/` bridge module. The mobile deployment builds the **same repo and branch** as the web deployment — one codebase, two isolated production instances (see the deployment topology section below). **The existing web/PWA Vercel project is not modified in any way.**

- **Why recommended:** the app is server-dependent (doc 1 §1) — bundling the frontend is a multi-day refactor, while the remote shell is near-zero frontend fork. Every web deploy updates both phones **instantly**, which is stronger than the stated requirement ("updates should not require manually reinstalling"). All 30+ modules keep working by construction — nothing is lost. Native wins (reliable push with DND/full-screen, exact alarms, NFC auto-open, stable storage) come from the plugin layer, not from rewriting UI.
- **Alternatives:** see the rejection table below.
- **Tradeoffs:** WebView rendering (acceptable — the UI is already a mobile-first PWA tuned on these exact phones); App Review compatibility deferred to Stage 2 (irrelevant on the chosen zero-review tracks); iOS offline cold-start needs the WKAppBoundDomains spike (doc 3 §2); web outage ⇒ native outage (identical to today's PWA exposure).
- **Recurring maintenance:** Capacitor major upgrade 1–2×/yr (~1h); binaries rebuilt only for plugin/icon changes, Play's yearly target-API ratchet, and TestFlight's 90-day expiry.
- **Paid services:** Apple $99/yr, Play $25 once, Firebase free.

### Stage 2 — bundled build target (deferred; the Unlisted-graduation chunk)

A second build target (`BUILD_TARGET=capacitor` → `output: "export"`) bundles the frontend into the shells; the Vercel deployment remains as pure API + web frontend. Scope (bounded, ~4–6 days):

1. Bearer-token fallback in the single auth helper `src/lib/supabase/server.ts` (accept `Authorization: Bearer <access_token>` alongside cookies — same pattern as the Wear OS brief).
2. CORS for the native origin, added centrally in `proxy.ts`.
3. Client-side auth gates replacing RSC `redirect()` for the exported tree; login via supabase-js with native storage.
4. `next/image` → `unoptimized` for the export target (1 file affected).
5. iOS privacy manifests + App Store assets (doc 4 §3).

**Why staged, not up-front:** Stage 2's only forcing function is Apple's guideline 4.2 (full review for Unlisted). TestFlight internal + Play internal have **no content review**, so deferring keeps the "never blocked by review" guarantee while shipping value in days instead of weeks. If the 90-day TestFlight ritual never annoys, Stage 2 may never be needed.

## Deployment topology — PWA isolation (hard constraint, added 2026-07-11)

> **Constraint (owner-stated): the PWA must never be interrupted or impacted by the native solution.** Isolation is enforced at two layers — infrastructure (this section) and code (the Non-Interference Contract below). Neither layer alone is sufficient.

### Two Vercel projects, one repo, one branch

| | `era-web` (existing project) | `era-mobile` (new project) |
|---|---|---|
| Source | this repo, `main` | this repo, `main` (same branch — **not** a separate `mobile` branch) |
| Domain | current web domain — **unchanged, binaries never reference it** | own domain (mobile subdomain or its `*.vercel.app` URL) — `server.url`, FCM links, and the shells pin **here** |
| Serves | browsers: laptop/tablet PWA, partner's Safari | both native shells only |
| Config | **never modified by this plan** | free to diverge (env, headers) if native ever needs it |
| Cron scheduler | ✅ **the only deployment the external scheduler targets** | ❌ never scheduled — same routes exist but must not be invoked (double execution = duplicate notifications/auto-posts; recurrence-safety hazard) |
| Rollback | independent (existing flow) | independent — mobile can be pinned/rolled back while web moves on, and vice versa |
| Env vars | as today | **full parity** with web (same Supabase, Gemini, Azure, VAPID, Firebase secrets) except `NEXT_PUBLIC_APP_URL` = mobile domain. Parity is a documented checklist item — drift causes confusing native-only bugs |
| Cost | — | $0 (second project on the same Vercel account) |

- **Why same branch, not a `mobile` branch:** a separate branch gives promotion control but guarantees staleness — the phones (the primary devices) would lag every daily web ship until someone remembers to merge. Same-branch keeps phones current automatically; isolation comes from independent rollback/pinning and the untouched web project.
- **Optional escalation dial:** during risky native bring-up windows, disable auto-promotion on `era-mobile` (Vercel per-project setting) so mobile production only moves on manual promote — web cadence unaffected either way. Default: auto-deploy both.
- **Shared-truth caveat (read this):** both deployments run the same code against the same Supabase DB. The deployment split does **not** protect the PWA from a bad shared-code change or a careless migration — the contract below and the additive-migration rule carry that half of the guarantee.
- Supabase Auth config: add the mobile domain to the redirect allow-list (password reset). Web and mobile sessions are independent Supabase sessions (different cookie jars) — expected, harmless; partner logs in once on the native app.

### PWA Non-Interference Contract (code-level invariants)

1. **Native code is invisible to browsers**: every native branch sits behind `isNative()` from `src/lib/native/` (dynamic imports, no-op on web); `@capacitor/*` never imported elsewhere.
2. **`public/sw.js`, `public/manifest.json`, and `public/manifests/*` are never edited for native reasons.** The browser push pipeline (VAPID/web-push) stays byte-identical for `platform='web'` rows.
3. **DB changes must be additive and backward-compatible** (the native-push migration is the template: new column with default, relaxed nullability — existing web rows and readers unaffected).
4. **API contract changes are union-extensions, never mutations** (subscribe route: the `web` variant of the discriminated union is today's schema, untouched).
5. **Every phase ends with the web-regression gate** (doc 5 testing table) before its exit criterion counts.
6. The only web-visible additions are inert to browsers: `.well-known` association files (fetched by OSes, not pages) and the vibrate→haptics shim (guarded to native iOS).
7. If any invariant must be broken, that is a **STOP condition** — surface it, don't proceed.

## Alternatives considered and rejected

| Alternative | Why rejected |
|---|---|
| **React Native / Expo rewrite** | Duplicates 30+ modules of UI; standing vault verdict "full native rewrite: not worth it, ever"; permanent double-maintenance for a 2-user app. No paid services saved. |
| **Flutter rewrite** | Same duplication problem plus a second language/toolchain. |
| **TWA / Bubblewrap (Android)** | Chrome container = no native plugin access — exact alarms, DND-bypass channels, full-screen intents, native NFC are all impossible. Android-only (no iOS story). Already rejected in the Wear OS brief. |
| **Stay PWA** | Fails the stated requirements: iOS web push has no interruption levels (no Time-Sensitive/DND handling), Safari can evict PWA storage, no NFC auto-open, no exact alarms. This was the status quo the request explicitly moves past. |
| **Bundled-first (skip remote shell)** | Front-loads the Stage-2 refactor before any native value ships; higher risk of breaking the web app; the only benefit (review-readiness) isn't needed on the chosen tracks. |
| **OneSignal / third-party push SaaS** | Unnecessary intermediary for 2 users; FCM v1 direct is free and ~150 lines given `googleapis` is already a dependency. |
| **Ionic Appflow (builds/OTA)** | Product sunset announced (ends 2027) — dead end. |
| **Single deployment, shells on a domain alias** | Cheaper (no second project) and solves domain-pinning, but no independent rollback/env/pause — fails the "PWA never impacted" constraint at the infrastructure layer. |
| **Separate `mobile` branch for the mobile project** | Strongest promotion control, but guarantees phone staleness against daily web ships; merge chore becomes a standing failure mode. Same-branch + optional manual promotion gives the control without the rot. |

## What stays unchanged (~98% of the repo)

- **Entire backend**: 186 API routes, Zod validation, household linking, error mapping.
- **Database**: schema, RLS, migrations workflow; cron routes + external scheduler.
- **AI layer** (Gemini, drafts/proposal pattern), Azure voice token proxy.
- **All React UI, hooks, contexts, feature dirs** — no component forks.
- **Offline stack**: queue, sync engine, persister, connectivity manager, safeFetch.
- **Realtime**: broadcast + postgres_changes (supabase-js works identically in WebViews).
- **Web push for browsers**: laptop/tablet keep VAPID web-push untouched.
- **Vercel deployment flow**, PWA manifests, hand-written `sw.js` for browser users.

## Web/mobile coexistence model

- **One codebase, two isolated production deployments.** Browsers get the PWA exactly as today from the untouched `era-web` project; phones run shells pointed at `era-mobile`; per-route PWA manifests remain for tablet/laptop installs.
- **Platform branching only via the bridge** (`isNativePlatform()` re-exported from `src/lib/native/`), never UA sniffing, never forked components.
- **Kill-switch by construction:** if a shell misbehaves, the phone falls back to the installed PWA immediately — zero downtime while fixing.
- **House rule to add to CLAUDE.md at implementation time:** *"UI/hooks never import `@capacitor/*` directly — always via `src/lib/native/` (dynamic import, no-op on web)."* Also: Feature Map + Atlas entries for the new surface, per Hard Rule 23.

## Repository & folder structure (target end state)

```
budget-app/
├─ capacitor.config.ts
├─ android/                  # committed native project (Gradle)
├─ ios/                      # committed native project (Xcode)
├─ resources/                # icon.png, splash.png sources
├─ public/.well-known/       # assetlinks.json, apple-app-site-association
├─ src/lib/native/           # bridge: index.ts (isNative), push.ts, deeplinks.ts,
│                            #   haptics.ts, notifications.ts, nfc.ts, geo.ts
└─ (everything else unchanged)
```

No monorepo split, no second repository — a small personal project stays maintainable by keeping the native layer as a thin, isolated directory pair plus one bridge module.
