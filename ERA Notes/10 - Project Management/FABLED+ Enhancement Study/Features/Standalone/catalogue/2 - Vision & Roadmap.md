# Catalogue · Vision & Roadmap

> [FABLED+ root](<../../../_index.md>) · **Catalogue** · [1 · Feature State](<1 - Feature State.md>) · [2 · Vision & Roadmap](<2 - Vision & Roadmap.md>) · [3 · Action Plan](<3 - Action Plan.md>) · [4 · Checklist](<4 - Checklist.md>)

## Enhancement thesis

Treat Catalogue as the household's versioned operating manual: reusable objects that declare compatibility, learn from instances, and never surprise existing records.

## Business and household value

Reliable reuse compounds capture speed and consistency across tasks, products, trips, and routines. Version discipline turns Catalogue from storage into household infrastructure.

Measure attention returned, conflict avoided, and outcomes improved—not engagement.

## Roadmap

1. Now — calculate template reuse and post-promotion edit deltas read-only.
2. Next — add semantic change classes and affected-instance preview.
3. Later — promote repeated successful instance edits back as reviewed template proposals.

## New opportunity set

### V1 — Template semantic versions

- **Mechanism:** Classify changes as cosmetic, default-only, compatible, or breaking and preserve the version used by each instance.
- **Smallest proof:** Version one frequently reused template locally.
- **Success measure:** A later change can be explained without altering existing instances.
- **Kill criterion:** Keep only change history if version classes add no decision value.
- **Invariant:** Existing instances never mutate implicitly.

### V2 — Instance drift review

- **Mechanism:** Compare promoted records with their source template and group repeated edits as feedback.
- **Smallest proof:** Inspect ten instances from one template.
- **Success measure:** One repeated edit becomes a useful template improvement proposal.
- **Kill criterion:** Stop if instances intentionally diverge with no stable pattern.
- **Invariant:** Drift is evidence, not an automatic merge.

### V3 — Household playbook pack

- **Mechanism:** Bundle related templates, prerequisites, roles, and instructions for repeatable household situations.
- **Smallest proof:** Build one guest-arrival or maintenance pack.
- **Success measure:** The pack replaces multiple manual setup steps without hiding actions.
- **Kill criterion:** Keep individual templates if bundling adds navigation.
- **Invariant:** Launching a pack produces reviewable proposals, not silent records.

## Existing-roadmap boundary

Trip template libraries and generic module manifests already exist; this pack addresses versioning, compatibility, and learning from reuse.

## Strategy guardrail

Start read-only or in shadow mode; earn persistence, notification, and automation.
