# Trips · Vision & Roadmap

> [FABLED+ root](<../../../_index.md>) · **Trips** · [1 · Feature State](<1 - Feature State.md>) · [2 · Vision & Roadmap](<2 - Vision & Roadmap.md>) · [3 · Action Plan](<3 - Action Plan.md>) · [4 · Checklist](<4 - Checklist.md>)

## Enhancement thesis

Treat a trip as a temporary branch of household operations: rehearse in isolation, commit with version checks, operate from contingency playbooks, and merge back with a receipt.

## Business and household value

Trips concentrate cost, schedule, and stress. Safe lifecycle orchestration can prevent large mistakes and demonstrate the app's cross-module advantage.

Measure attention returned, risk reduced, and outcomes improved—not engagement.

## Roadmap

1. Now — snapshot and replay the cascade against fixtures with exact preconditions.
2. Next — implement a branch-like dry run and one contingency playbook.
3. Later — learn from trip completion/debrief outcomes without generalizing a new modes engine.

## New opportunity set

### V1 — Trip branch

- **Mechanism:** Build intended side effects against a versioned snapshot, detect drift, then apply or rebase explicitly.
- **Smallest proof:** Dry-run one historical trip activation in fixtures.
- **Success measure:** Every effect and conflict is known before live commit.
- **Kill criterion:** Use impact preview plus version check if full branch abstraction is excessive.
- **Invariant:** Dry run has no mutation access; commit is idempotent.

### V2 — Contingency playbook

- **Mechanism:** Predeclare actions/data for delay, cancellation, lost item, health issue, or budget shock.
- **Smallest proof:** Create one delay and one lost-item playbook as proposals.
- **Success measure:** A simulated disruption produces a useful response in under two minutes.
- **Kill criterion:** Keep a static emergency note if actions are not used.
- **Invariant:** Emergency suggestions never move money or cancel plans automatically.

### V3 — Trip decision register

- **Mechanism:** Track shared choices, owner, deadline, evidence, approval, and supersession across planning.
- **Smallest proof:** Use it for transport, lodging, and one activity.
- **Success measure:** Current decisions are clear without searching chat.
- **Kill criterion:** Use pinned thread decisions if register adds ceremony.
- **Invariant:** Read receipt is not agreement.

## Existing-roadmap boundary

Impact panel, per-cascade opt-out, re-entry briefing, trip rollup, template library, smart packing, modes engine, and capture-from-wild are existing ideas; this pack adds branch isolation and contingency.

## Strategy guardrail

Start read-only or shadowed; use existing proposal and mutation owners; earn automation.
